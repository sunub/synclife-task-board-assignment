## 1. 상태 구조
- 서버 상태와 클라이언트(낙관적) 상태를 어떻게 나눴나요? 어디서 관리하나요?

Task의 최종 기준은 서버에 두고, 서버에서 받은 데이터는 **TanStack Query cache**에서 관리했습니다.

Query cache에는 API 응답인 `Task[]`를 그대로 저장하지 않고, `byId`와 `idsByStatus`로 구성된 `TaskBoardModel`로 정규화했습니다. 보드에서는 Task를 ID로 조회하거나 특정 컬럼으로 이동시키는 작업이 반복되기 때문에, 이 구조를 사용하면 전체 목록을 다시 탐색하지 않고 관련 Task와 컬럼만 변경할 수 있습니다. 실패 시에도 전체 상태가 아니라 해당 Task만 복구할 수 있어 다른 변경에 미치는 영향을 줄일 수 있습니다.

낙관적 상태는 별도의 전역 store에 복제하지 않고 **동일한 Query cache에 임시로 반영**했습니다. 같은 Task를 Query cache와 별도 store에 동시에 보관하면 성공 응답, 실패 rollback, 충돌 처리마다 두 상태의 동기화를 관리해야 하기 때문입니다. 사용자가 Task를 이동하거나 수정하면 Query cache를 먼저 변경해 화면에 즉시 반영하고, 성공하면 서버 응답으로 확정했습니다.

실패 시 복구 기준은 작업 성격에 따라 다르게 두었습니다. 생성은 임시 Task를 제거하고, 수정과 삭제는 요청 전에 저장한 Task를 기준으로 복구했습니다. 반면 이동은 같은 Task에 대해 여러 요청이 빠르게 겹칠 수 있으므로 단순히 요청 직전의 Task로 되돌리지 않았습니다. 연속 이동 중 요청 직전 상태는 이미 클라이언트가 낙관적으로 반영한 중간 상태일 수 있기 때문입니다.

이를 위해 이동 작업에 한해서 Task별로 마지막으로 확인된 서버 확정 상태를 관리했습니다. 화면에 보이는 낙관적 상태와 서버에서 확인된 상태를 구분함으로써, 모든 이동 요청이 실패하면 중간 낙관 상태가 아니라 마지막 서버 확정 상태로 복구할 수 있습니다. 또한 중간 이동 요청이 서버에서 성공한 뒤 마지막 요청만 실패한 경우에는, 최초 상태가 아니라 마지막 성공 상태로 복구할 수 있습니다.

검색어나 정렬 기준은 서버 데이터가 아니라 화면을 제어하는 클라이언트 상태로 분리했습니다. 다만 검색 결과나 정렬된 Task 목록 자체는 별도 상태로 저장하지 않고, `TaskBoardModel`과 현재 화면 조건을 조합해 파생했습니다. 이를 통해 같은 Task 데이터를 여러 곳에 중복 저장하지 않고, 검색이나 정렬이 바뀌더라도 서버 상태를 담은 Query cache의 구조는 유지했습니다.

- 이렇게 잡은 이유와, 고려했다가 버린 대안은?

이 프로젝트에서는 상태를 저장 위치보다 **최종 기준을 누가 가지는가**에 따라 구분했습니다. Task는 화면에서 이동하거나 수정할 수 있지만, 요청 실패나 다른 변경과의 충돌이 발생할 수 있으므로 클라이언트의 값이 항상 확정된 상태라고 볼 수 없습니다.

낙관적 업데이트를 하면 사용자 입장에서는 빠르게 반응하는 것처럼 보이지만, 실제 요청은 실패할 수도 있고 409 충돌이 날 수도 있습니다. 그래서 클라이언트 상태를 서버와 완전히 분리된 또 하나의 원천처럼 만들면, 성공/실패/충돌 상황에서 어느 쪽을 기준으로 맞춰야 하는지가 복잡해진다고 봤습니다.

이 결정을 내리기 이전에 버린 대안은 Zustand 을 사용한 상태 관리였습니다. 

처음에는 Zustand 같은 별도 store도 고려했습니다. status별로 task를 나눠 들고 있으면 렌더링 제어나 상태 분리는 쉬워 보였습니다. 하지만 서버 상태는 이미 TanStack Query가 관리하고 있는데, 같은 task 데이터를 Zustand에도 들고 있으면 두 저장소를 계속 동기화해야 합니다. 특히 rollback, 409 충돌, 서버 성공 응답 반영 같은 규칙을 양쪽에 맞춰야 해서 단순한 상태 분리가 아니라 복잡도가 늘어나는 방향이라고 판단하여 선택하지 않았습니다.


## 2. 낙관적 업데이트 & 롤백
- 이동/수정/삭제/생성 시 UI를 먼저 반영하고 실패하면 어떻게 되돌리나요?

이 작업에서는 사용자가 액션을 했을 때 서버 응답을 기다린 뒤 화면을 바꾸기보다, 먼저 TanStack Query cache를 수정해서 UI에 즉시 반영했습니다. 보드는 사용자가 드래그, 수정, 삭제 같은 조작을 반복하는 화면이기 때문에 서버 응답을 기다린 뒤에야 UI가 바뀌면 조작감이 크게 떨어진다고 판단했습니다.

다만 낙관적으로 반영한 값은 최종 확정 상태가 아닙니다. 요청은 실패할 수 있고, 다른 변경과 충돌할 수도 있습니다. 그래서 중요한 기준은 “어떻게 빨리 바꿀 것인가”뿐 아니라 “실패했을 때 어떤 상태가 진짜로 믿을 수 있는 상태인가”였습니다.

생성은 서버에 아직 존재하지 않는 Task를 화면에 먼저 보여주는 작업입니다. 따라서 실패했을 때 되돌릴 이전 Task가 없습니다. 서버가 생성을 받아들이지 않았다면 그 Task는 실제로 존재한 적이 없는 것이므로, 임시 id로 추가한 Task를 cache에서 제거하는 방식이 가장 자연스럽다고 봤습니다.

수정은 이미 존재하는 Task의 내용을 바꾸는 작업입니다. 수정 요청이 실패했다는 것은 서버가 사용자의 변경 내용을 받아들이지 않았다는 뜻이므로, 요청 직전의 Task가 가장 안전한 복구 기준입니다. 그래서 수정 전 `previousTask`를 저장해 두고, 실패하면 그 Task를 다시 cache에 반영했습니다.

삭제는 수정과 반대 방향입니다. 화면에서는 먼저 Task를 제거하지만, 서버 삭제가 실패했다면 실제 데이터는 삭제되지 않은 것입니다. 이 경우 사용자가 봐야 하는 상태는 삭제 직전의 Task이므로, 삭제 전에 저장한 `previousTask`를 다시 추가해 복원했습니다.

이동은 수정이나 삭제보다 더 조심해야 했습니다. 같은 Task를 짧은 시간 안에 여러 컬럼으로 연속 이동할 수 있기 때문입니다. 이때 요청 직전의 `previousTask`는 서버가 확정한 상태가 아니라, 이전 이동을 클라이언트가 낙관적으로 반영한 중간 상태일 수 있습니다. 예를 들어 `todo -> in-progress -> done`을 빠르게 수행했는데 모든 이동 요청이 실패했다면, 서버는 여전히 `todo`를 가지고 있습니다. 그런데 마지막 요청의 `previousTask`인 `in-progress`로 되돌리면 실패한 낙관 상태가 화면에 남게 됩니다.

그래서 이동 실패 시에는 요청 직전 상태가 아니라, 마지막으로 확인된 서버 확정 상태인 Confirmed Task State를 기준으로 복원했습니다. 이 기준을 두면 모든 이동 요청이 실패한 경우에는 최초 서버 확정 상태로 돌아가고, 중간 이동 요청이 성공한 뒤 마지막 요청만 실패한 경우에는 마지막으로 서버가 받아들인 상태로 돌아갈 수 있습니다.

- 롤백을 "서버 재조회" vs "이전 스냅샷 복원" 중 무엇으로 했고 그 이유는?

기본 rollback은 서버 재조회나 전체 보드 스냅샷 복원이 아니라, 실패한 액션과 관련된 Task만 복원하는 방식으로 했습니다.

서버 재조회를 기본으로 쓰지 않은 이유는 실패 범위에 비해 비용이 너무 크기 때문입니다. 현재 API는 Task 하나만 다시 가져오는 구조가 아니라 전체 Task 목록을 가져오는 구조입니다. 카드 하나의 이동이나 수정이 실패할 때마다 전체 데이터를 다시 가져오고, 다시 파싱하고, 다시 `TaskBoardModel`로 정규화하면 작은 실패 하나가 큰 작업으로 번집니다.

전체 `TaskBoardModel` 스냅샷을 저장했다가 통째로 복원하는 방식도 선택하지 않았습니다. 이 방식은 구현은 단순하지만, 실패한 요청과 무관한 정상 변경까지 되돌릴 수 있습니다. 예를 들어 한 Task의 이동 요청이 실패하기 전에 다른 Task의 수정이나 생성이 이미 성공했다면, 오래된 전체 스냅샷으로 복원하는 순간 그 정상 변경까지 사라질 수 있습니다.

그래서 rollback의 범위를 실패한 Task 하나로 좁혔습니다. 생성은 실제 서버 데이터가 생기지 않았으므로 임시 Task를 제거하고, 수정과 삭제는 요청 직전의 Task가 안전한 기준이므로 `previousTask`를 복원했습니다. 이동은 연속 요청 때문에 요청 직전 상태를 믿을 수 없으므로 Confirmed Task State를 복원했습니다.

이 방식은 서버를 최종 기준으로 유지하면서도, 실패 처리 비용을 줄이고, 실패한 액션과 무관한 다른 Task의 정상 변경을 보호하기 위한 선택이었습니다.

## 3. 경쟁 상태(race condition)
- 같은 카드를 빠르게 연속 이동할 때 최종 상태가 어긋나지 않도록 무엇을 했나요?
  (요청 취소 / 버전·순번 비교 / last-write-wins 등)

사용자가 같은 Task를 짧은 시간 안에 여러 Board Column으로 이동하면, 요청을 보낸 순서와 응답이 도착하는 순서가 달라질 수 있습니다.

예를 들어 사용자가 Task를 `todo → in-progress → done`으로 빠르게 이동했을 때, 첫 번째 요청의 성공이나 실패 응답이 두 번째 요청보다 늦게 도착할 수 있습니다. 이 응답을 도착 순서대로 cache에 반영하면 오래된 응답이 사용자의 마지막 이동 결과를 덮어쓸 수 있습니다.

### 요청 취소만으로 해결하지 않은 이유

처음에는 새로운 이동이 시작될 때 이전 요청을 취소하는 방식을 고려했습니다.

하지만 클라이언트에서 요청을 취소하더라도, 이미 서버에 도착한 요청은 처리되었을 수 있습니다. 이 경우 서버의 Task 상태와 version은 변경되지만, 클라이언트는 해당 결과를 알지 못할 수 있습니다.

따라서 요청 취소만으로는 서버와 클라이언트의 상태 불일치를 막기 어렵다고 판단했습니다.

### Task별 Sequence로 오래된 응답 무시

각 Task에는 client-side sequence를 부여했습니다.

이동 요청이 시작될 때마다 해당 Task의 sequence를 증가시키고, 성공·실패·409 응답이 돌아왔을 때 요청 당시의 sequence와 현재 sequence를 비교했습니다.

두 값이 일치하는 경우에만 응답을 처리하고, 일치하지 않으면 이미 더 최신 이동 요청이 발생한 것으로 판단해 다음 작업을 모두 건너뜁니다.

* 서버 응답의 cache 반영
* 낙관적 업데이트 rollback
* 409 충돌 안내 및 재시도

이를 통해 오래된 응답이 사용자의 마지막 이동 의도를 덮어쓰지 못하도록 했습니다.

### `previousTask`가 아닌 서버 확정 상태로 Rollback

Sequence는 어떤 응답이 최신 요청에 대한 것인지는 알려주지만, 최신 요청이 실패했을 때 어느 상태로 복원해야 하는지는 알려주지 않습니다.

일반적인 낙관적 업데이트처럼 요청 직전의 `previousTask`로 복원하면, 연속 이동 상황에서 잘못된 상태로 돌아갈 수 있습니다.

예를 들어 다음 이동이 빠르게 발생했다고 가정합니다.

```text
todo → in-progress → done
```

두 번째 요청이 시작될 때 UI에는 첫 번째 이동이 낙관적으로 반영되어 있으므로, 두 번째 요청의 `previousTask`는 `in-progress`입니다.

이 상태에서 두 요청이 모두 실패했는데 두 번째 요청의 `previousTask`로 복원하면 Task는 `in-progress`에 남습니다. 그러나 서버에서는 어떤 이동도 성공하지 않았으므로 올바른 상태는 최초 서버 확정 상태인 `todo`입니다.

이를 해결하기 위해 Task ID별로 **Confirmed Task State**를 관리했습니다.

첫 번째 이동 요청이 시작될 때 현재 Task를 서버 확정 상태로 저장하고, 같은 Task에 대한 후속 이동 요청은 이 값을 덮어쓰지 않습니다.

이후 서버에서 이동 요청이 성공하면 응답으로 받은 Task를 새로운 Confirmed Task State로 저장합니다. 409 응답에서 서버의 최신 Task인 `current`를 받은 경우에도 이를 새로운 서버 확정 상태로 사용합니다.

최신 이동 요청이 실패하면 요청 직전의 `previousTask`가 아니라 Confirmed Task State로 복원합니다.

따라서 다음 두 경우를 모두 올바르게 처리할 수 있습니다.

* 모든 이동 요청이 실패하면 최초 서버 확정 상태로 복원
* 중간 요청은 성공하고 마지막 요청만 실패하면 마지막 성공 상태로 복원

### 409 충돌은 서버 상태를 기준으로 Rebase 후 재시도

409 Conflict는 일반적인 네트워크 실패와 다르게 처리했습니다.

최신 이동 요청이 오래된 version을 기준으로 서버에 도착하면 서버는 409 응답과 함께 현재 Task 상태인 `current`를 반환합니다.

이 경우 다음 순서로 처리합니다.

1. `current`를 cache와 Confirmed Task State에 반영합니다.
2. 클라이언트와 서버의 version 기준을 일치시킵니다.
3. 사용자가 마지막으로 선택한 target status와 `current.status`를 비교합니다.
4. 두 상태가 다르면 `current.version`을 사용해 같은 이동을 한 번 더 요청합니다.

즉, 서버 상태를 최신 version으로 다시 맞춘 뒤 사용자의 마지막 이동 의도를 재적용하는 rebase retry 방식입니다.

### 정리

경쟁 상태를 처리하기 위해 세 가지 기준을 함께 사용했습니다.

1. **Sequence**
   오래된 응답이 사용자의 최신 이동 결과를 덮어쓰지 못하게 합니다.
2. **Version**
   요청이 서버의 최신 Task 상태를 기준으로 수행되고 있는지 검증합니다.
3. **Confirmed Task State**
   최신 요청이 실패했을 때 중간 낙관 상태가 아니라 마지막 서버 확정 상태로 복원합니다.

이를 통해 서버의 데이터 정합성은 version으로 보장하고, 클라이언트에서 사용자의 마지막 이동 의도는 sequence와 rebase retry로 유지했습니다.

## 4. 대량 데이터 성능 (5,000개)
- 검색/필터/드래그를 부드럽게 만들기 위해 무엇을 선택했고 **무엇을 포기**했나요?
- 가능하면 측정 근거(예: 최적화 전/후 체감·프로파일)를 남겨 주세요.

## 4. 대량 데이터 성능 최적화 — 5,000개 Task

### 수행한 것

성능 최적화에 앞서 비용을 **프론트엔드에서 줄일 수 있는 비용**과 **현재 API 구조에서는 줄이기 어려운 비용**으로 구분했습니다.

현재 `GET /api/tasks`는 5,000개의 Task를 한 번에 반환합니다. 따라서 초기 네트워크 전송량과 JSON 파싱 비용은 프론트엔드만으로 근본적으로 줄이기 어렵다고 판단했습니다. 이에 목표를 초기 로딩 시간 단축보다, 데이터 로드 이후 검색·정렬·스크롤·드래그 과정에서 발생하는 **DOM 생성 및 렌더링 비용 감소**에 두었습니다.

각 Board Column의 Card 목록에는 `@tanstack/react-virtual`을 적용했습니다. 전체 데이터와 스크롤 높이는 유지하면서, viewport와 overscan 범위에 포함된 약 51개의 Card만 실제 DOM에 렌더링했습니다. 이를 통해 Task가 5,000개이더라도 모든 Card가 동시에 DOM에 생성되지 않도록 했습니다.

가상화 과정에서 가장 중요하게 고려한 부분은 제목 길이에 따라 달라지는 Card의 동적 높이였습니다. `measureElement`나 `getBoundingClientRect()`로 실제 높이를 측정할 수 있지만, 스크롤 중 DOM 측정과 위치 보정이 반복되면 Layout 계산이 증가하고 layout thrashing이 발생할 가능성이 있습니다.

이를 줄이기 위해 실제 DOM을 측정하는 대신 `@chenglou/pretext`로 Card 제목의 높이를 렌더링 전에 예측했습니다. Canvas 기반 텍스트 측정을 통해 제목의 줄 수와 높이를 계산하고, 그 값을 TanStack Virtual의 초기 높이 추정값으로 사용했습니다. 그 결과, 렌더링 이후 실제 높이를 다시 측정하고 Card 위치를 보정하는 횟수를 줄일 수 있었습니다.

Card 제목이 한 줄에서 최대 세 줄까지 변하는 환경에서 비교한 결과, Pretext를 적용한 방식은 스크롤 중 발생한 `Layout` 및 `UpdateLayoutTree` 비용을 초당 기준 약 23% 줄였습니다. 애플리케이션 코드와 직접 관련된 Layout 작업만 비교했을 때도 약 14% 감소했습니다.

Layout 작업을 완전히 제거한 것은 아니지만, 실제 높이 확인 이후 반복되던 위치 보정과 순간적으로 크게 발생하던 Layout 비용을 완화했다는 점에서 의미 있는 개선으로 판단했습니다.

### 포기한 것과 트레이드오프

가상화를 적용하면서 모든 Card를 미리 DOM에 렌더링하는 단순한 스크롤 구조는 포기했습니다.

기존 방식은 초기 렌더링 비용과 DOM 유지 비용이 크지만, 스크롤 중에는 이미 생성된 요소를 그대로 사용할 수 있습니다. 반면 가상화 방식은 스크롤할 때마다 가시 영역을 계산하고, 화면에 들어오는 Card를 mount하고 벗어나는 Card를 unmount해야 합니다. 이 과정에서 Card 위치 계산, React Commit, 동적 높이 처리와 같은 추가 연산이 발생합니다.

측정 결과는 다음과 같습니다.

| 지표                |      기존 방식 |       가상화 적용 |         변화 |
| ----------------- | ---------: | -----------: | ---------: |
| 실제 렌더링 Card 수     |   약 5,000개 |        약 51개 |   약 99% 감소 |
| DOM node 수        |  약 80,120개 | 중앙값 약 1,048개 | 약 98.7% 감소 |
| 초당 Paint 비용       | 약 14.3ms/s |    약 5.7ms/s |   약 60% 감소 |
| Main thread 점유율   |      2.70% |        9.81% |         증가 |
| Dropped Frame     |         0건 |          15건 |         증가 |
| 50ms 이상 Long Task |         0건 |           0건 |         동일 |

가상화를 통해 실제 렌더링 Card 수는 약 99%, DOM node 수는 약 98.7% 감소했으며, 초당 Paint 비용도 약 60% 줄었습니다. 브라우저가 지속적으로 관리하고 그려야 하는 렌더링 트리의 규모를 크게 축소한 결과입니다.

반면 스크롤 중 가시 영역 계산과 Card mount·unmount가 반복되면서 Main thread 점유율은 2.70%에서 9.81%로 증가했고, Dropped Frame도 15건 발생했습니다. 다만 두 방식 모두 50ms 이상의 Long Task는 발생하지 않았습니다. 즉, Main thread가 장시간 차단된 것이 아니라 짧은 가상화 연산이 반복적으로 추가된 형태였습니다.

따라서 이번 최적화는 모든 성능 비용을 줄인 작업이 아닙니다. 대신 데이터 수에 비례해 증가하는 **대규모 DOM 유지 비용**을, viewport 범위에 한정된 **작은 증분 연산 비용**으로 전환했습니다.

현재 스크롤 경로에는 추가 개선 여지가 있지만, 데이터가 증가해도 실제 DOM 규모를 일정하게 유지할 수 있다는 점에서 기존 방식보다 확장 가능한 구조를 확보했습니다.

## 5. 정답이 없던 결정들 (명세서에 명시되지 않은 항목)
- **네트워크 완전 단절** 시 동작을 어떻게 정의했나요? 왜?

네트워크가 끊어진 상태에서는 보드를 **현재 화면에 있는 데이터를 조회할 수는 있지만, 새 쓰기 작업은 수행하지 않는 Read-only 상태**로 정의했습니다. 

이 결정을 한 이유는 낙관적 업데이트의 사용자 경험 문제를 줄이기 위해서입니다. 이 보드는 카드 이동이나 수정 시 먼저 화면을 바꾸고, 이후 서버 요청 결과에 따라 성공 또는 롤백을 처리합니다. 그런데 명확히 오프라인인 상태에서도 조작을 허용하면 사용자는 “이동된 것처럼 보였다가 곧바로 원래 상태로 돌아가는” 흐름을 반복해서 겪게 됩니다. 이는 실제로는 저장되지 않은 동작을 성공처럼 보여주는 것이기 때문에, 차라리 오프라인 상태에서는 쓰기 조작을 즉시 막는 편이 더 예측 가능하다고 판단했습니다.

구현은 온라인 상태를 상위에서 한 번만 구독한 뒤, 이를 `editable` / `read-only` 보드 모드로 변환하는 방식으로 처리했습니다. `Board`에서는 이 모드를 기준으로 카드 드래그, 수정, 생성, 저장, 삭제 가능 여부를 각각의 boolean 값으로 파생해 하위 컴포넌트에 전달했습니다.

쓰기 mutation은 모두 `networkMode: "online"`을 유지하고, read-only 상태에서는 UI 차단과 mutation 내부 guard를 통해 실행되지 않도록 방어했습니다.

다만 이번 구현은 Service Worker, IndexedDB, 요청 큐를 이용한 완전한 오프라인 퍼스트 기능은 아닙니다. 새로고침 이후에도 오프라인 데이터를 유지하거나, 오프라인 중 발생한 쓰기 작업을 나중에 동기화하는 범위까지는 포함하지 않았습니다. 현재 단계에서는 데이터 동기화 복잡도를 키우기보다, 네트워크 단절 시 사용자가 실패할 작업을 반복하지 않도록 보드의 쓰기 동작을 명확히 제한하는 데 초점을 맞췄습니다.

- **409 충돌** 시 사용자에게 무엇을 보여주고 어떻게 해소하나요? 왜?

409 충돌은 작업 종류별로 다르게 처리했습니다. 핵심 기준은 “사용자의 마지막 의도를 안전하게 다시 적용할 수 있는가”였습니다.

이동은 `status` 하나를 바꾸는 동작이고, 사용자의 의도도 마지막으로 드롭한 컬럼으로 비교적 명확합니다. 그래서 최신 이동 요청이 409를 받으면 서버가 내려준 `current` task를 먼저 cache에 반영해 version 기준을 맞춘 뒤, 사용자가 마지막으로 원한 target status를 `current.version`으로 한 번만 자동 재시도했습니다. 재시도가 성공하면 사용자는 별도 충돌 메시지 없이 마지막으로 드롭한 컬럼에 task가 남아 있는 결과를 보게 됩니다. 이 처리는 같은 카드를 빠르게 여러 번 옮겼을 때, 이전 요청이 서버 version을 먼저 올려도 마지막 이동 의도가 사라지지 않게 하기 위한 것입니다.

반면 수정은 자동 재시도하지 않았습니다. 수정에는 `title`, `description`, `priority`, `status`처럼 여러 필드가 함께 들어갈 수 있고, 서버의 최신 값과 사용자가 입력 중인 draft를 어떤 기준으로 합칠지 명확한 merge 정책이 필요합니다. 이 상태에서 단순히 최신 version으로 다시 PATCH하면 다른 사용자의 변경이나 서버 최신 값을 덮어쓸 수 있습니다. 그래서 수정 409에서는 서버의 `current` task를 cache에 반영해 화면의 기준 상태는 최신화하되, panel은 닫지 않아 사용자가 입력한 draft는 유지하고 충돌 메시지를 보여주도록 했습니다.

일반 네트워크 실패나 500 에러는 자동 재시도하지 않았습니다. 클라이언트 입장에서는 실패 원인이 일시적인지, 서버가 실제로 요청을 처리했는지, 같은 요청을 다시 보내도 안전한지 확실히 알 수 없기 때문입니다. 따라서 이동/수정/삭제는 실패한 작업 단위로 이전 상태를 복원하고, 생성은 optimistic하게 추가했던 임시 task를 제거했습니다. 이렇게 전체 목록을 되돌리거나 무조건 재조회하지 않고, 실패한 사용자 액션의 영향 범위만 되돌리도록 했습니다.

- 실패한 쓰기의 **재시도**: 자동 vs 수동 — 무엇을 택했고 이유는?

일반적인 쓰기 실패에 대해서는 자동 재시도를 기본 정책으로 두지 않았습니다. 생성, 수정, 삭제, 이동 요청이 실패하면 먼저 낙관적으로 바꿨던 화면을 되돌리고 사용자에게 실패를 알려주는 방식을 선택했습니다. 요청이 실패한 원인이 일시적인 서버 오류인지, 사용자가 다시 판단해야 하는 상황인지 알 수 없는데 자동으로 계속 재시도하면 화면 상태와 서버 상태를 더 헷갈리게 만들 수 있다고 봤습니다.

다만 409 충돌은 일반적인 500 실패와 다르게 다뤘습니다. 특히 이동에서는 사용자의 마지막 이동 의도가 명확하고, 서버가 최신 Task와 version을 내려주는 경우가 있기 때문에, 최신 요청에 한해서 서버 최신 version 위에 마지막 이동 의도를 한 번 더 반영하려는 흐름을 두었습니다. 반대로 수정 저장에서는 사용자의 draft와 서버 최신 값이 충돌할 수 있으므로 자동으로 덮어쓰기보다, 서버 최신 상태를 반영하고 draft를 보존한 뒤 사용자에게 충돌을 알리는 쪽을 선택했습니다.

정리하면, 일반 실패는 자동 재시도보다 rollback과 사용자 피드백을 우선했습니다. 409는 서버 정합성을 먼저 회복하되, 이동처럼 사용자의 마지막 의도가 단순한 경우에만 제한적으로 다시 맞춰보는 방식을 택했습니다.

## 6. 그 외 트레이드오프 / 남은 한계
- 시간 제약으로 포기한 것, 알고 있는 버그·한계, 더 있었다면 했을 것.

**컬럼 내부 수동 정렬 지원 포기 및 카드 이동 후 위치 파악 문제**

이번 과제에서 시간 제약과 API 구조상 한계로 구현하지 못한 기능은 **동일 컬럼 내 드래그 앤 드롭을 통한 수동 정렬**입니다. 

초기에는 사용자가 카드를 원하는 위치에 드롭하여 순서를 변경할 수 있도록 고려했습니다. 하지만 설계 논의 과정(`CONTEXT.md` 및 `docs` 하위 문서 참고)에서 다음과 같은 한계를 확인했습니다.

1. **상태 영속화 한계:** 현재 서버의 데이터 모델에는 상태값만 있고 사용자가 지정한 순서를 저장할 필드(예: `position`, `rank`)가 없습니다. 프론트엔드에서 임시로 순서를 변경할 수는 있지만, 서버에 저장되지 않아 새로고침이나 다중 탭 환경에서 순서가 초기화됩니다. 이는 사용자에게 데이터가 온전히 저장되었다는 잘못된 인식을 줄 수 있습니다.
2. **기존 데이터 필드의 오용:** 데이터의 마지막 수정 시각을 의미하는 `updatedAt` 필드를 수동 정렬 기준으로 활용하는 것도 배제했습니다. 수정 시각을 화면 표시 우선순위 용도로 임의 조정하면 본래의 데이터 의미가 훼손되기 때문입니다.
3. **가상 스크롤 환경에서의 추가 복잡도:** 가상 스크롤에서도 수동 정렬 구현은 가능하지만, 화면 밖 항목의 삽입 위치 표현, 자동 스크롤, 측정 캐시 갱신, 롤백 시 순서 복원까지 함께 고려해야 합니다. 서버에 정렬 순서를 저장할 필드가 없는 상황에서 이 복잡도를 감수해도 영속적인 사용자 경험을 보장하기 어렵다고 판단했습니다.


이러한 이유로 드래그 앤 드롭은 **다른 컬럼으로 상태를 변경**하는 기능으로만 한정했고, 특정 위치 삽입을 유도하는 UI는 제외했습니다. 컬럼 내부의 정렬은 자동 정렬 정책(`updatedAt` -> `createdAt` -> `id` 순)을 일관되게 따르도록 구현했습니다.

**남은 아쉬움과 향후 개선 방향:**
수동 정렬을 제외하면서 남은 아쉬운 점은 **컬럼 간 이동 후 카드의 위치를 파악하기 어렵다**는 점입니다. 사용자가 카드를 다른 컬럼으로 옮겼을 때 원하는 위치에 지정할 수 없으므로, 도착한 컬럼의 자동 정렬 기준에 따라 카드가 임의로 배치되어 직관적으로 찾기 어려울 수 있습니다. 이를 보완하기 위해 카드가 이동된 위치를 간접적으로 확인할 수 있도록 처리했지만, 사용자 경험 측면에서는 여전히 개선의 여지가 있습니다.

실제 서비스 환경이고 시간이 충분했다면, 프론트엔드에서 임시로 상태를 관리하기보다는 **서버 모델에 정렬 전용 필드를 추가하는 방향으로 백엔드와 협의**했을 것입니다. 이를 통해 서버 기반의 안정적인 수동 정렬을 지원하고, 다중 사용자 환경에서의 데이터 충돌 처리 및 재정렬 정책을 함께 설계하여 더 나은 보드 경험을 제공했을 것입니다.

**완전한 오프라인 지원(오프라인 퍼스트) 적용의 한계**

이번 마일스톤에서는 "네트워크 완전 단절" 시의 오프라인 대응을 **'현재 세션 내에서의 Read-only 및 조작 잠금(레벨 1)'** 수준으로 한정했습니다. 브라우저의 `navigator.onLine` 및 API 에러를 감지하여 즉각적으로 드래그 앤 드롭 등 보드 조작을 잠그고 오프라인 상태임을 지속적으로 노출함으로써, 사용자가 불필요한 낙관적 업데이트 롤백을 겪지 않도록 UX를 개선했습니다.

하지만 사용자가 오프라인 상태에서 브라우저를 **새로고침(F5)할 경우 빈 화면이나 접속 에러를 보게 되는 한계**가 남아있습니다. 시간이 더 있었다면 Service Worker를 활용한 정적 에셋 캐싱과 `IndexedDB`를 통한 TanStack Query 데이터 영구 저장(Persist)을 도입하여, 오프라인 환경에서 새로고침을 하거나 앱에 재진입해도 이전 데이터를 조회할 수 있는 완전한 오프라인 퍼스트(PWA) 경험을 제공했을 것입니다. 데이터 동기화 복잡도를 고려하여 이번 작업에서는 제외했습니다.
