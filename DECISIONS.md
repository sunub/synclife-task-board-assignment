## 1. 상태 구조
- 서버 상태와 클라이언트(낙관적) 상태를 어떻게 나눴나요? 어디서 관리하나요?

이 프로젝트의 task 데이터는 사용자가 화면에서 계속 조작하지만, 최종 기준은 서버에 있어야 한다고 봤습니다. 카드 이동이나 수정은 실패할 수도 있고, 다른 변경과 충돌할 수도 있기 때문에 클라이언트가 가진 값이 항상 정답이라고 볼 수는 없었습니다.

그래서 저는 상태를 “어디에 저장하느냐”보다 “누가 최종 기준이냐”를 기준으로 나눴습니다. 서버에서 온 task 데이터는 TanStack Query cache에서 관리했고, 서버 응답을 그대로 쓰지는 않았습니다. 보드에서는 특정 task를 id로 찾고, status별 column을 렌더링하고, 실패했을 때 한 task만 되돌리는 일이 중요했기 때문에 칸반 보드내에서 자주 사용되고 구분되는 속성을 기준으로 byId, idsByStatus 형태의 TaskBoardModel로 바꿔 저장했습니다.

낙관적 업데이트는 별도 store에 따로 복제하지 않았습니다. 사용자가 카드를 옮기면 같은 Query cache를 먼저 바꿔서 화면에 즉시 반영하고, 서버 요청이 성공하면 서버가 돌려준 값으로 확정했습니다. 일반 실패는 이동 전에 저장해 둔 task만 되돌렸습니다. 409 충돌은 mutation 종류에 따라 다르게 처리했습니다. 이동 요청에서는 서버가 내려준 최신 task를 기준점으로 반영한 뒤, 사용자의 마지막 target status가 아직 서버에 반영되지 않았다면 서버의 최신 version으로 한 번 더 재시도했습니다. 수정 요청에서는 서버 최신 task를 cache에 반영하되, 사용자가 입력하던 draft는 보존하고 충돌을 알렸습니다.

클라이언트에서 필요한 값 중 일부는 서버 데이터에서 계산할 수 있는 파생 데이터라고 봤습니다. 예를 들어 검색어가 적용된 task 목록이나 현재 정렬 기준에 따른 표시 목록은 서버가 따로 저장해야 하는 데이터가 아니라, 이미 받아온 task 데이터와 사용자의 화면 조건을 조합해서 계산할 수 있는 값입니다. 그래서 이런 값들을 별도 상태로 저장하지 않고, TaskBoardModel의 byId, idsByStatus를 기준으로 필요한 순간에 계산하도록 했습니다. 이렇게 하면 같은 데이터를 여러 곳에 중복 저장하지 않아도 되고, 검색어나 정렬값이 바뀌어도 서버 상태 cache 자체를 오염시키지 않을 수 있습니다.

- 이렇게 잡은 이유와, 고려했다가 버린 대안은?

이 구조를 선택한 가장 큰 이유는 서버 데이터가 최종 기준이어야 했기 때문입니다. 낙관적 업데이트를 하면 사용자 입장에서는 빠르게 반응하는 것처럼 보이지만, 실제 요청은 실패할 수도 있고 409 충돌이 날 수도 있습니다. 그래서 클라이언트 상태를 서버와 완전히 분리된 또 하나의 원천처럼 만들면, 성공/실패/충돌 상황에서 어느 쪽을 기준으로 맞춰야 하는지가 복잡해진다고 봤습니다.

이 결정을 내리기 이전에 버린 대안은 Zustand 을 사용한 상태 관리였습니다. 

처음에는 Zustand 같은 별도 store도 고려했습니다. status별로 task를 나눠 들고 있으면 렌더링 제어나 상태 분리는 쉬워 보였습니다. 하지만 서버 상태는 이미 TanStack Query가 관리하고 있는데, 같은 task 데이터를 Zustand에도 들고 있으면 두 저장소를 계속 동기화해야 합니다. 특히 rollback, 409 충돌, 서버 성공 응답 반영 같은 규칙을 양쪽에 맞춰야 해서 단순한 상태 분리가 아니라 복잡도가 늘어나는 방향이라고 판단했습니다.

그래서 최종적으로는 서버 상태의 기준은 TanStack Query cache에 두고, cache 내부 구조만 보드 사용 방식에 맞게 바꾸는 쪽을 선택했습니다. 별도 store를 두지 않고 TaskBoardModel로 정규화해서 단일 task 접근, column별 렌더링, task 단위 rollback을 처리했습니다. 성공 경로에서는 전체 재조회 대신 서버가 반환한 task만 setQueryData로 반영했습니다. 이 방식이 서버를 최종 기준으로 유지하면서도 낙관적 업데이트와 대량 데이터 처리 비용을 함께 줄일 수 있는 선택이라고 봤습니다.


## 2. 낙관적 업데이트 & 롤백
- 이동/수정/삭제/생성 시 UI를 먼저 반영하고 실패하면 어떻게 되돌리나요?

이 작업에서는 사용자가 액션을 했을 때 서버 응답을 기다린 뒤 화면을 바꾸기보다, 먼저 TanStack Query cache를 수정해서 UI에 즉시 반영했습니다. 다만 이 값은 최종 확정된 값이 아니라 “서버 요청이 성공할 것이라고 예상한 임시 상태”로 봤습니다.

이동과 수정은 요청 전에 기존 task를 previousTask로 저장해 둡니다. 그 다음 cache에는 이동된 status나 수정된 내용을 먼저 반영하고, 서버 요청이 성공하면 서버가 반환한 task로 다시 확정했습니다. 일반 실패가 나면 저장해 둔 previousTask를 다시 cache에 반영해서 해당 task만 이전 상태로 되돌렸습니다. 409 충돌처럼 서버가 최신 task를 내려주는 경우에는 무조건 previousTask로 되돌리지 않았습니다. 이동 요청의 409에서는 서버 최신 task를 기준점으로 맞춘 뒤 사용자의 마지막 target status를 최신 version으로 한 번 재시도했고, 수정 요청의 409에서는 서버 최신 task를 반영하면서 사용자의 draft를 보존했습니다.

삭제는 먼저 cache에서 task를 제거해서 화면에서 바로 사라지게 했고, 서버 삭제가 실패하면 삭제 전에 저장해 둔 task를 다시 추가해서 복원했습니다. 생성은 기존 task가 없기 때문에 임시 id를 가진 task를 먼저 cache에 넣고, 서버 생성이 성공하면 임시 task를 서버가 반환한 실제 task로 교체했습니다. 실패하면 그 임시 task만 제거했습니다.

- 롤백을 "서버 재조회" vs "이전 스냅샷 복원" 중 무엇으로 했고 그 이유는?

기본 rollback은 서버 재조회가 아니라, 요청 직전에 저장해 둔 이전 상태를 task 단위로 복원하는 방식으로 했습니다. 여기서 중요한 점은 전체 보드 스냅샷을 통째로 되돌린 것이 아니라, 실패한 액션과 관련된 task만 되돌렸다는 점입니다.

서버 재조회를 기본으로 쓰지 않은 이유는 현재 API가 task 하나만 다시 가져오는 구조가 아니라 전체 task 목록을 가져오는 구조이기 때문입니다. 카드 하나를 이동하거나 수정하는 데 실패했을 때마다 전체 데이터를 다시 fetch하고, parse하고, TaskBoardModel로 다시 정규화하는 건 비용이 크다고 봤습니다. 전체 스냅샷을 되돌리는 것을 기본 성공/실패 경로로 두면 단일 변경에도 전체 데이터 처리가 반복되는 문제가 있었습니다.

그렇다고 정규화된 전체 TaskBoardModel 스냅샷을 저장했다가 통째로 복원하는 방식도 선택하지 않았습니다. 네트워크 응답 순서가 늦어지는 동안 다른 task 수정이나 다른 낙관적 업데이트가 성공했을 수 있는데, 전체 스냅샷을 되돌리면 그런 정상 변경까지 덮어쓸 수 있기 때문입니다.

그래서 최종적으로는 previousTask를 저장해 두고 실패한 task만 복원하는 방식을 선택했습니다. 이 방식이 서버를 최종 기준으로 유지하면서도, 불필요한 전체 재조회 비용을 줄이고, rollback 범위도 사용자 액션 단위로 좁힐 수 있다고 판단했습니다. 단, 409 충돌은 단순 실패 rollback으로 보지 않았습니다. 서버가 내려준 current task를 먼저 반영해 version 기준을 맞추고, 이동 요청이라면 마지막 이동 의도를 current.version으로 한 번 재시도했습니다. 재시도까지 충돌하거나, 수정 요청처럼 자동 병합 기준이 분명하지 않은 경우에는 서버 최신 상태를 반영하고 사용자에게 충돌을 알렸습니다.

## 3. 경쟁 상태(race condition)

- 같은 카드를 빠르게 연속 이동할 때 최종 상태가 어긋나지 않도록 무엇을 했나요?
  (요청 취소 / 버전·순번 비교 / last-write-wins 등)

사용자가 카드를 빠르게 옮기면, 요청이 보낸 순서대로 응답이 돌아온다는 보장이 없기 때문에 첫 번째 요청이 나중에 성공하거나 실패하면서 두 번째 이동 결과를 덮어쓸 수 있어 단순히 서버 응답이 올 때마다 cache에 반영하는 방식은 위험하다고 봤습니다.

그래서 떠오른 해결책 중, 처음에 생각할 수 있는 방법은 요청을 취소하는 것이었지만, 이동 요청은 이미 서버에 도달했을 수 있고 서버의 version이 변경될 수도 있습니다. 그래서 클라이언트에서 이전 요청을 취소하는 것만으로는 충분하지 않다고 판단했습니다. 서버의 version도 사용했지만, 이 값은 서버 입장에서 “현재 task 버전과 요청 버전이 맞는지” 확인하는 동시성 제어용이지, 클라이언트 안에서 어떤 이동이 사용자의 마지막 의도인지 알려주지는 않습니다.

그래서 task id별 sequence를 두었습니다. 이동이 시작될 때마다 해당 task의 sequence를 증가시키고, 성공/실패/409 응답이 돌아왔을 때 현재 최신 sequence와 일치하는 응답만 처리했습니다. 일치하지 않는 응답은 오래된 응답으로 보고 cache 반영, rollback, conflict 안내를 모두 건너뛰었습니다.

다만 sequence만으로는 충분하지 않았습니다. sequence는 “이 응답이 사용자의 최신 이동 요청에 대한 응답인가”는 알려주지만, 최신 요청이 오래된 version으로 서버에 도착해 409를 받는 상황까지 해결하지는 못합니다. 예를 들어 첫 번째 이동 요청이 서버에서 먼저 성공해 version을 올리고, 두 번째 이동 요청이 이전 version으로 409를 받으면, 서버 current를 그대로 반영하는 순간 마지막 이동 의도가 사라질 수 있습니다.

그래서 최신 이동 요청의 409에서는 서버 current를 기준점으로 cache에 반영한 뒤, 사용자의 마지막 target status가 current.status와 다르면 current.version으로 같은 이동을 한 번 더 PATCH했습니다. 이렇게 해서 오래된 성공/실패/409는 sequence로 무시하고, 최신 요청의 version 충돌은 rebase retry로 처리했습니다. 결과적으로 서버 정합성은 version으로 확인하고, 클라이언트의 최신 이동 의도는 sequence와 409 rebase retry를 함께 사용해 보존했습니다.

## 4. 대량 데이터 성능 (5,000개)
- 검색/필터/드래그를 부드럽게 만들기 위해 무엇을 선택했고 **무엇을 포기**했나요?

대량 데이터 성능에서 먼저 구분한 것은 “줄일 수 있는 비용”과 “줄일 수 없는 비용”이었습니다. 현재 API는 GET /api/tasks로 5,000개 Task를 한 번에 내려주기 때문에 초기 fetch와 JSON parse 비용은 프론트엔드만으로 줄이기 어렵다고 봤습니다. 그래서 최적화 목표를 초기 로딩 단축이 아니라, 데이터가 로드된 이후 검색, 정렬, 스크롤, 드래그 중 발생하는 DOM/render 비용을 줄이는 쪽으로 잡았습니다.

이를 위해 각 Board Column 내부의 Card 목록을 @tanstack/react-virtual로 독립 가상화했습니다. Board 전체를 하나의 virtual list로 만들 수도 있었지만, Column은 Task Status별 독립 영역이고 count, empty state, drop target, scroll state를 각각 가져야 했기 때문에 Column별 가상화를 선택했습니다.

가상 스크롤을 적용하면서 가장 신경 쓴 부분은 동적 Card 높이를 어떻게 계산할지였습니다. measureElement나 getBoundingClientRect로 렌더된 Card 높이를 계속 측정하면, 스크롤 중 DOM read/write가 반복되어 layout thrashing이 생길 수 있다고 봤습니다.

그래서 런타임 기본 경로에서는 DOM 측정 대신 @chenglou/pretext를 사용해 제목 텍스트 높이를 미리 계산하고, 그 값을 TanStack Virtual의 estimateSize에 넘겼습니다. 계산 결과는 task title과 card width 기준으로 cache해서 같은 조건에서는 재사용했습니다. 대신 실제 DOM 높이를 매번 보정하는 방식은 포기했고, Card 높이 변동 원인을 제목 텍스트와 고정된 layout 상수 안으로 제한했습니다.

- 가능하면 측정 근거(예: 최적화 전/후 체감·프로파일)를 남겨 주세요.


## 5. 정답이 없던 결정들 (명세서에 명시되지 않은 항목)
- **네트워크 완전 단절** 시 동작을 어떻게 정의했나요? 왜?

네트워크가 완전히 끊어졌을 때는 보드를 **'현재 세션 내 조회만 가능한 상태(Read-only)'**로 전환하고 카드 이동 등의 조작을 즉각적으로 잠그도록 정의했습니다.

그 이유는 낙관적 업데이트(Optimistic Update)의 부작용을 방지하기 위해서입니다. 오프라인 상태임에도 조작을 허용하면, 사용자가 카드를 옮길 때마다 화면에서는 성공한 것처럼 보였다가 API 실패로 인해 곧바로 원래 자리로 롤백(Rollback)되는 현상이 무한 반복되어 사용자에게 큰 불쾌감을 줍니다.

따라서 브라우저의 `navigator.onLine` 이벤트와 API 요청 실패를 결합해 오프라인 상태(Lie-Fi 포함)를 감지하고, 일시적인 스낵바가 아닌 지속적으로 노출되는 UI로 오프라인 상태임을 알리며 조작을 차단하는 방식을 택했습니다. 단, 이번 마일스톤의 제약과 데이터 동기화 복잡도를 고려하여 Service Worker와 IndexedDB를 활용한 완전한 오프라인 퍼스트(새로고침 후에도 유지되는 상태)까지는 구현하지 않았습니다.

- **409 충돌** 시 사용자에게 무엇을 보여주고 어떻게 해소하나요? 왜?

409 충돌은 모든 쓰기에서 같은 방식으로 처리하지 않았습니다. 이동은 변경 범위가 `status` 하나로 좁고, 사용자의 마지막 의도가 명확합니다. 그래서 최신 이동 요청이 409를 받으면 서버가 내려준 current task를 cache에 반영해 version 기준을 맞춘 뒤, 사용자가 마지막으로 원한 target status를 current.version으로 한 번 자동 재시도했습니다. 이 재시도가 성공하면 사용자는 마지막으로 드롭한 컬럼에 task가 남아 있는 결과를 보게 됩니다.

반면 수정은 자동 재시도하지 않았습니다. 수정에는 title, description, priority, status처럼 여러 필드가 포함될 수 있고, 서버 최신 값과 사용자가 입력한 draft를 어떻게 합칠지 별도 merge 정책이 필요합니다. 이 경우 무리하게 자동 재시도하면 다른 변경을 덮어쓸 수 있다고 봤습니다. 그래서 수정 409에서는 서버 current를 cache에 반영하고, panel의 사용자 draft는 보존한 채 충돌 메시지를 보여주도록 했습니다.

일반 네트워크 실패나 500 에러는 자동 재시도하지 않았습니다. 실패 원인이 일시적인지, 서버가 실제로 처리했는지, 사용자가 같은 작업을 다시 시도해도 되는지 클라이언트가 확실히 알 수 없기 때문입니다. 대신 이동/수정/삭제/생성 모두 실패한 작업 단위로 rollback하거나 임시 task를 제거하고 사용자에게 실패를 알렸습니다.

- 실패한 쓰기의 **재시도**: 자동 vs 수동 — 무엇을 택했고 이유는?

일반적인 쓰기 실패에 대해서는 자동 재시도를 기본 정책으로 두지 않았습니다. 생성, 수정, 삭제, 이동 요청이 실패하면 먼저 낙관적으로 바꿨던 화면을 되돌리고 사용자에게 실패를 알려주는 방식을 선택했습니다. 요청이 실패한 원인이 일시적인 서버 오류인지, 사용자가 다시 판단해야 하는 상황인지 알 수 없는데 자동으로 계속 재시도하면 화면 상태와 서버 상태를 더 헷갈리게 만들 수 있다고 봤습니다.

다만 409 충돌은 일반적인 500 실패와 다르게 다뤘습니다. 특히 이동에서는 사용자의 마지막 이동 의도가 명확하고, 서버가 최신 Task와 version을 내려주는 경우가 있기 때문에, 최신 요청에 한해서 서버 최신 version 위에 마지막 이동 의도를 한 번 더 반영하려는 흐름을 두었습니다. 반대로 수정 저장에서는 사용자의 draft와 서버 최신 값이 충돌할 수 있으므로 자동으로 덮어쓰기보다, 서버 최신 상태를 반영하고 draft를 보존한 뒤 사용자에게 충돌을 알리는 쪽을 선택했습니다.

정리하면, 일반 실패는 자동 재시도보다 rollback과 사용자 피드백을 우선했습니다. 409는 서버 정합성을 먼저 회복하되, 이동처럼 사용자의 마지막 의도가 단순한 경우에만 제한적으로 다시 맞춰보는 방식을 택했습니다.

## 6. 그 외 트레이드오프 / 남은 한계
- 시간 제약으로 포기한 것, 알고 있는 버그·한계, 더 있었다면 했을 것.

**컬럼 내부 수동 정렬 지원 포기 및 카드 이동 후 위치 파악 문제**

이번 과제에서 시간 제약과 API 구조상 한계로 구현하지 못한 기능은 **동일 컬럼 내 드래그 앤 드롭을 통한 수동 정렬**입니다. 

초기에는 사용자가 카드를 원하는 위치에 드롭하여 순서를 변경할 수 있도록 고려했습니다. 하지만 설계 논의 과정(`CONTEXT.md` 및 `docs` 하위 문서 참고)에서 다음과 같은 한계를 확인했습니다.

1. **상태 영속화 한계:** 현재 서버의 데이터 모델에는 상태값만 있고 사용자가 지정한 순서를 저장할 필드(예: `position`, `rank`)가 없습니다. 프론트엔드에서 임시로 순서를 변경할 수는 있지만, 서버에 저장되지 않아 새로고침이나 다중 탭 환경에서 순서가 초기화됩니다. 이는 사용자에게 데이터가 온전히 저장되었다는 잘못된 인식을 줄 수 있습니다.
2. **기존 데이터 필드의 오용:** 데이터의 마지막 수정 시각을 의미하는 `updatedAt` 필드를 수동 정렬 기준으로 활용하는 것도 배제했습니다. 수정 시각을 화면 표시 우선순위 용도로 임의 조정하면 본래의 데이터 의미가 훼손되기 때문입니다.
3. **가상 스크롤 환경의 렌더링 비용:** 대량의 데이터를 처리하기 위해 가상 스크롤을 도입한 상태에서, 화면 밖에 있는 카드들 사이의 정확한 삽입 위치를 계산하고 화면에 표시하는 작업은 렌더링 비용을 크게 증가시킵니다.

이러한 이유로 드래그 앤 드롭은 **다른 컬럼으로 상태를 변경**하는 기능으로만 한정했고, 특정 위치 삽입을 유도하는 UI는 제외했습니다. 컬럼 내부의 정렬은 자동 정렬 정책(`updatedAt` -> `createdAt` -> `id` 순)을 일관되게 따르도록 구현했습니다.

**남은 아쉬움과 향후 개선 방향:**
수동 정렬을 제외하면서 남은 아쉬운 점은 **컬럼 간 이동 후 카드의 위치를 파악하기 어렵다**는 점입니다. 사용자가 카드를 다른 컬럼으로 옮겼을 때 원하는 위치에 지정할 수 없으므로, 도착한 컬럼의 자동 정렬 기준에 따라 카드가 임의로 배치되어 직관적으로 찾기 어려울 수 있습니다. 이를 보완하기 위해 카드가 이동된 위치를 간접적으로 확인할 수 있도록 처리했지만, 사용자 경험 측면에서는 여전히 개선의 여지가 있습니다.

실제 서비스 환경이고 시간이 충분했다면, 프론트엔드에서 임시로 상태를 관리하기보다는 **서버 모델에 정렬 전용 필드를 추가하는 방향으로 백엔드와 협의**했을 것입니다. 이를 통해 서버 기반의 안정적인 수동 정렬을 지원하고, 다중 사용자 환경에서의 데이터 충돌 처리 및 재정렬 정책을 함께 설계하여 더 나은 보드 경험을 제공했을 것입니다.

**완전한 오프라인 지원(오프라인 퍼스트) 적용의 한계**

이번 마일스톤에서는 "네트워크 완전 단절" 시의 오프라인 대응을 **'현재 세션 내에서의 Read-only 및 조작 잠금(레벨 1)'** 수준으로 한정했습니다. 브라우저의 `navigator.onLine` 및 API 에러를 감지하여 즉각적으로 드래그 앤 드롭 등 보드 조작을 잠그고 오프라인 상태임을 지속적으로 노출함으로써, 사용자가 불필요한 낙관적 업데이트 롤백을 겪지 않도록 UX를 개선했습니다.

하지만 사용자가 오프라인 상태에서 브라우저를 **새로고침(F5)할 경우 빈 화면이나 접속 에러를 보게 되는 한계**가 남아있습니다. 시간이 더 있었다면 Service Worker를 활용한 정적 에셋 캐싱과 `IndexedDB`를 통한 TanStack Query 데이터 영구 저장(Persist)을 도입하여, 오프라인 환경에서 새로고침을 하거나 앱에 재진입해도 이전 데이터를 조회할 수 있는 완전한 오프라인 퍼스트(PWA) 경험을 제공했을 것입니다. 데이터 동기화 복잡도를 고려하여 이번 작업에서는 제외했습니다.
