# AI 도구 활용 (AI_USAGE)

> AI 사용은 **권장**됩니다. 이 문서는 "얼마나 썼는지"가 아니라
> **AI가 준 답을 어떻게 검증·판단했는지**를 봅니다. 솔직하게 적어 주세요.

## 1. 사용한 도구

Codex, Antigravity

## 3. AI 답을 그대로 쓰지 않고 검증·거부·수정한 사례 (중요)

### 1. 409 충돌에서 모든 mutation을 자동 재시도하지 않도록 수정

처음에는 409 충돌이 발생하면 서버의 최신 version을 받아 요청을 다시 보내는 방식이 합리적으로 보였습니다. 하지만 이를 모든 mutation에 동일하게 적용하면 오히려 서버의 최신 변경을 덮어쓸 수 있다고 판단했습니다. 이동과 수정은 사용자의 의도가 명확한 정도가 다르기 때문입니다.

이동은 주로 `status` 하나를 바꾸는 동작이며, 사용자가 마지막으로 드롭한 column이라는 의도가 비교적 명확합니다. 따라서 최신 이동 요청에서 409가 발생한 경우에만 서버의 `current.version`을 기준으로 마지막 target status를 한 번 더 적용하도록 했습니다.

반면 수정은 `title`, `description`, `priority`, `status`가 함께 바뀔 수 있습니다. 이 경우 사용자의 draft를 자동으로 재전송하면 서버에서 먼저 반영된 최신 변경을 덮어쓸 가능성이 있습니다. 그래서 수정 409에서는 자동 재시도를 하지 않고, 서버의 최신 task를 cache에 반영하되 사용자가 작성 중이던 draft는 side panel에 보존하도록 했습니다.

이 결정은 자동 재시도가 항상 좋은 UX는 아니라는 판단에서 나왔습니다. 의도가 단순한 이동은 재시도를 통해 마지막 의도를 보존할 수 있지만, 병합 기준이 불분명한 수정은 사용자가 최신 상태를 확인한 뒤 다시 판단하도록 하는 편이 안전하다고 봤습니다.

### 2. 전체 스냅샷 rollback 대신 task 단위 rollback으로 수정

낙관적 업데이트 실패 시 이전 상태로 되돌린다는 설명은 자연스러웠지만, 여기서 “이전 상태”를 전체 보드 스냅샷으로 관리하는 것은 위험하다고 판단했습니다.

사용자가 여러 task를 빠르게 조작하는 동안 하나의 요청이 실패했을 때 전체 스냅샷을 복원하면, 해당 요청 이후에 성공한 다른 task의 변경까지 함께 덮어쓸 수 있기 때문입니다. rollback 자체가 또 다른 데이터 정합성 문제를 만들 수 있는 구조였습니다.

그래서 rollback의 기준을 “실패한 사용자 액션이 변경한 범위만 되돌릴 수 있는가”로 잡았습니다. 이동이나 수정 실패는 해당 task 하나만 복원하고, 생성 실패는 임시 task만 제거하며, 삭제 실패는 삭제 전 task만 다시 추가하도록 설계했습니다.

결과적으로 mutation 직전에 전체 보드를 저장하지 않고, 해당 작업에 필요한 `previousTask`만 저장했습니다. 일반 실패가 발생하면 그 task만 복원하고, 다른 task의 변경에는 영향을 주지 않도록 했습니다.

단, 409 충돌은 단순한 요청 실패가 아니라 서버에 더 최신 version이 존재하는 상황입니다. 따라서 이전 task로 rollback하지 않고 서버가 내려준 `current` task를 먼저 반영한 뒤, 이동과 수정의 의도에 따라 서로 다른 충돌 정책을 적용했습니다.

### 3. `invalidateQueries` 중심 설계를 `TaskBoardModel` 기반 갱신으로 수정

처음에는 카드 이동이나 수정이 성공한 뒤 `invalidateQueries(['tasks'])`를 호출해 전체 목록을 다시 가져오는 방식이 가장 안전해 보였습니다. 서버 데이터를 다시 조회하면 클라이언트와 서버의 상태를 쉽게 일치시킬 수 있기 때문입니다.

하지만 README와 mock API를 확인해 보니, 현재 API는 column별 조회가 아니라 `GET /api/tasks`에서 전체 task 목록을 반환하는 구조였습니다. 이 환경에서 단일 task가 변경될 때마다 5,000개의 데이터를 다시 fetch하고, JSON을 parse하고, 보드 구조로 normalize하는 것은 변경 범위에 비해 비용이 과하다고 판단했습니다.

또한 PATCH 성공 응답으로 서버가 수정된 task 하나를 반환하고 있었습니다. 서버가 확정한 변경 결과를 이미 받은 상황에서, 그 task 하나를 반영하기 위해 전체 목록을 다시 요청할 필요는 없다고 봤습니다.

그래서 AI가 제안한 재조회 중심 흐름을 그대로 사용하지 않고, 서버 응답 `Task[]`를 `byId`, `idsByStatus` 형태의 `TaskBoardModel`로 변환해 Query cache에 저장했습니다. 성공 경로에서는 `invalidateQueries` 대신 `setQueryData`를 사용해 서버가 반환한 단일 task만 cache에 반영했습니다.

이를 통해 특정 task의 조회와 수정은 `byId`에서 처리하고, column 간 이동은 관련된 `idsByStatus`만 갱신할 수 있었습니다. 서버 상태의 최종 기준은 유지하면서도, 단일 변경이 전체 데이터 재처리로 이어지지 않도록 범위를 제한했습니다.

### 4. Card 고정 높이 대신 동적 높이와 Pretext 기반 계산으로 수정

가상 스크롤에서는 Card 높이를 고정하는 방식이 구현과 스크롤 위치 계산 면에서 유리해 보였습니다. 하지만 이 보드에서 Card는 단순한 시각 요소가 아니라 사용자가 업무 내용을 확인하는 정보 단위라고 판단했습니다.

긴 task 제목을 고정 높이에 맞춰 잘라내면 렌더링 계산은 단순해지지만, 사용자가 task의 핵심 내용을 확인하기 어려워집니다. 그래서 성능 최적화가 핵심 정보의 노출을 해쳐서는 안 된다는 기준으로 고정 높이 방식을 채택하지 않았습니다.

대신 제목은 임의로 줄 수를 제한하지 않고, 텍스트 길이에 따라 Card 높이가 달라지도록 했습니다. 다만 동적 높이를 위해 스크롤 중 모든 요소를 `getBoundingClientRect`나 `measureElement`로 측정하면 layout 계산 비용이 증가할 수 있습니다.

이를 피하기 위해 런타임 기본 경로에서는 DOM을 직접 측정하지 않고, `@chenglou/pretext`로 제목의 예상 높이를 사전에 계산해 `@tanstack/react-virtual`의 `estimateSize`에 전달했습니다. Card 높이에 영향을 주는 요소도 제목 텍스트와 고정된 layout 상수로 제한해 계산 결과가 실제 높이와 최대한 일치하도록 했습니다.

DOM 측정 fallback은 기본 경로에서 제외했습니다. 이를 통해 긴 제목의 정보 접근성을 유지하면서도, 동적 높이 Card가 가상 스크롤의 반복적인 layout 측정으로 이어지지 않도록 설계했습니다.

### 5. 더블클릭 inline edit 대신 side panel 단일 편집 모델을 선택

AI와 초기 설계를 논의할 때, 카드 제목을 빠르게 수정할 수 있도록 더블클릭 기반 inline edit을 적용하는 방안을 검토했습니다. 일반적인 목록 UI에서는 별도의 화면 이동 없이 값을 수정할 수 있다는 점에서 충분히 유효한 방식이었습니다.

하지만 현재 프로젝트의 Card는 단순한 텍스트 블록이 아니라 드래그 대상이면서 가상 스크롤의 item입니다. Card 내부에 input과 draft 상태를 추가하면 클릭과 드래그의 구분, 입력 요소의 focus 관리, 편집 중 Card 높이 변경에 따른 virtualization 재계산이 하나의 컴포넌트 안에서 결합될 수 있다고 판단했습니다.

그래서 AI가 제안한 inline edit을 그대로 적용하지 않고, Card는 task 정보 표시와 드래그 역할에 집중하도록 했습니다. 편집은 명시적인 수정 버튼을 통해 열리는 side panel form에서만 수행하도록 제한했습니다.

이 방식은 카드에서 즉시 제목을 수정하는 편의성은 다소 줄어듭니다. 대신 보드 조작과 form 조작의 책임을 분리할 수 있고, draft와 validation, 서버 오류, 409 충돌 상태를 하나의 편집 영역에서 일관되게 관리할 수 있다는 장점이 더 크다고 판단했습니다.

## 4. 스스로 판단해서 결정한 부분

### 1. 컬럼 내부 수동 정렬은 이번 구현 범위에서 제외

AI와 일반적인 칸반 보드 UX를 검토하는 과정에서는 카드를 컬럼 내부의 원하는 위치에 드롭하는 수동 정렬도 자연스러운 기능으로 논의되었습니다.

그러나 현재 Task 모델에는 컬럼 내부 순서를 영속적으로 저장할 position, rank, order와 같은 필드가 없습니다. 프론트엔드에서 배열의 순서만 임시로 변경하면 현재 화면에서는 정렬된 것처럼 보이지만, 새로고침하거나 서버 데이터를 다시 조회하는 순간 순서가 사라집니다.

이는 사용자에게 저장된 기능처럼 보이지만 실제로는 서버에 보존되지 않는 동작을 제공하는 것이므로, 올바른 기능이라고 보기 어렵다고 판단했습니다. updatedAt을 정렬 기준처럼 변경하는 방법도 검토할 수 있었지만, 수정 시각을 사용자 정의 순서를 표현하는 필드로 오용하면 데이터의 의미가 훼손됩니다.

따라서 AI와 검토한 일반적인 수동 정렬 기능은 이번 범위에서 제외했습니다. 대신 현재 API가 명확히 저장할 수 있는 컬럼 간 이동, 즉 task의 status 변경만 제공하기로 결정했습니다.

기능의 수를 늘리는 것보다 서버에 영속화할 수 있고 동작 의미가 명확한 기능만 제공하는 편이 더 중요하다고 판단했습니다.

### 2. 성능 최적화 목표를 초기 네트워크 로딩보다 로드 이후 조작감으로 우선순위 결정

AI와 대량 데이터 처리 방식을 논의할 때, 페이지네이션이나 점진적 데이터 요청처럼 초기 로딩 비용을 줄이는 방향도 검토했습니다.

하지만 현재 과제에서는 API 구조를 변경할 수 없고, GET /api/tasks가 전체 task를 한 번에 반환합니다. 이 조건에서는 프론트엔드만으로 초기 네트워크 전송량 자체를 줄일 수 없습니다. 따라서 실제로 줄이지 못한 비용을 초기 로딩 최적화라고 설명하는 것은 적절하지 않다고 판단했습니다.

그래서 AI가 제시한 일반적인 대량 데이터 최적화 방향을 그대로 적용하지 않고, 현재 프론트엔드가 실질적으로 개선할 수 있는 영역으로 성능 목표를 재설정했습니다.

제가 집중한 대상은 데이터가 로드된 이후의 스크롤, 검색, 드래그, 렌더링 조작감이었습니다. 이를 위해 column별 가상화를 적용하고, task 데이터를 TaskBoardModel로 정규화했으며, mutation 결과도 전체 목록이 아닌 단일 task 단위로 cache에 반영했습니다.

초기 데이터 수신량을 줄였다고 포장하기보다, 주어진 API 제약 안에서 사용자가 반복적으로 체감하는 렌더링과 인터랙션 비용을 줄이는 것이 더 타당하다고 판단했습니다.

### 3. 수정 성공 후 invalidateQueries 대신 PATCH 응답으로 단일 task cache를 확정

AI와 서버 상태 동기화 방식을 논의할 때, 수정 요청이 성공한 뒤 invalidateQueries를 실행해 task 목록 전체를 다시 조회하는 방식도 검토했습니다. 일반적인 서버 상태 관리에서는 mutation 이후 재조회를 통해 최신 상태를 보장할 수 있다는 장점이 있습니다.

그러나 현재 API는 task 하나를 다시 조회하는 endpoint를 제공하지 않고, GET /api/tasks를 호출하면 전체 task 목록을 반환합니다. 따라서 task 하나를 수정할 때마다 전체 데이터를 다시 요청하고, JSON을 parse하고, TaskBoardModel 형태로 normalize하는 것은 대량 데이터 환경에서 불필요한 비용이라고 판단했습니다.

또한 PATCH 성공 응답이 서버에 최종 저장된 최신 task를 반환하므로, 성공 결과를 확정하기 위해 전체 목록을 다시 요청할 필요도 없다고 봤습니다.

그래서 AI가 제시한 전체 재조회 방식 대신, 성공 응답으로 받은 task를 applyServerTask를 통해 기존 TaskBoardModel cache에 직접 반영했습니다. 이를 통해 서버 응답을 최종 기준으로 유지하면서도, 수정되지 않은 수천 개의 task를 다시 가져오고 가공하는 비용을 피할 수 있었습니다.

### 4. 더블클릭 inline edit 대신 side panel 단일 편집 모델을 선택

AI와 초기 설계를 논의할 때, 카드 제목을 빠르게 수정할 수 있도록 더블클릭 기반의 inline edit을 적용하는 방안을 검토했습니다. 일반적인 목록 UI에서는 별도의 화면 이동 없이 값을 수정할 수 있다는 점에서 충분히 유효한 방식이었습니다.

그러나 현재 프로젝트의 Card는 단순한 텍스트 블록이 아니라 드래그 대상이면서 가상 스크롤의 item입니다. 카드 내부에 input과 draft 상태를 추가하면 클릭과 드래그의 구분, 입력 요소의 focus 관리, 편집 중 카드 높이 변경에 따른 virtualization 측정이 하나의 컴포넌트 안에서 결합될 수 있다고 판단했습니다.

따라서 AI가 제시한 inline edit 방식을 그대로 채택하지 않고, Card는 task 정보 표시와 드래그 역할에 집중시키기로 했습니다. 편집은 명시적인 수정 버튼을 통해 열리는 side panel form에서만 수행하도록 결정했습니다.

이 방식은 카드에서 즉시 제목을 수정하는 편의성은 다소 줄어들지만, 보드 조작과 form 조작의 책임을 분리하고 편집 상태를 하나의 영역에서 관리할 수 있다는 장점이 더 크다고 판단했습니다.
